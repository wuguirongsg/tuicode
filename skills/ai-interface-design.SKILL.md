# AI 界面设计 Skill

> **核心思想**：AI 产品体验不是"在模型上包一层 UI"——设计本身决定了 AI 系统能否真正被人所用。

---

## 一、统一分析框架：ICO 三阶段

所有 AI 交互都可拆解为三个阶段，每阶段有独立的失败模式：

| 阶段 | 定义 | 核心设计问题 |
|------|------|------------|
| **Input（输入）** | 用户意图进入系统的时刻 | 如何帮用户准确表达意图？ |
| **Computation（计算）** | 系统处理/路由/生成的黑箱阶段 | 如何让等待可理解、错误可恢复？ |
| **Output（输出）** | 结果返回给用户的时刻 | 如何让输出可信、可用、可操作？ |

**诊断口诀**：遇到任何 AI 界面问题，先问——"这是 Input、Computation 还是 Output 阶段的问题？"

## 二、输入设计

### 2.1 意图的三个通道（同时存在）

| 通道 | 定义 | 设计应对 |
|------|------|---------|
| **隐式上下文** (Implicit Context) | 系统从环境推断，用户未主动提供 | 显示系统"正在读取的内容"，给用户纠正机会 |
| **显式提示** (Explicit Prompting) | 用户输入的文字/语音 | 引导写好提示词，降低对提示工程的认知负担 |
| **直接操控** (Direct Manipulation) | 按钮、滑块、选中、手势 | 遵循 Jakob's Law——延续用户已有操作习惯 |

**原则**：三个通道共同分担意图传达的负担，减少用户需要显式说明的内容。

### 2.2 CARE 提示框架

引导用户写出高质量提示词的四要素：

| 要素 | 含义 | 弱 vs 强示例 |
|------|------|------------|
| **C – Context** | 我是谁？场景是什么？ | "写封邮件" → "我是PM，需告知保守型金融客户延期3周..." |
| **A – Action** | 具体操作是什么？ | "帮我看数据" → "分析季度数据，识别增长品类、找低于目标的地区、提3条策略" |
| **R – Results** | 输出形式、长度、面向谁？ | "摘要" → "2页执行摘要，结论导向，适合C级高管汇报" |
| **E – Examples** | 期望风格的参考样本 | 提供1-2段理想范例，胜过大量文字描述 |

> **产品侧实践**：通过结构化表单、模板预填、示例提示词等 UI 手段，帮用户自然地提供 CARE 四要素，无需学习"提示词工程"。

### 2.3 输入 UI 设计模式

- **允许 Markdown 格式化**：可提升模型准确率约 20%（GPT-4.1 法律任务测试数据）
- **结构化输入**（下拉+文本混合）：固定场景降低歧义
- **混合模式**（预设按钮 + 开放输入）：兼顾效率与灵活性
- **图像输入**：延续 Photoshop 类用户习惯（套索选区、marching ants），不发明新习语

**两个主要风险**：
- **讨好偏差 (Sycophancy)**：模型顺从用户假设 → 可设计"换个角度"按钮，高风险场景触发异见提示
- **长度-精度悖论**：提示词过长，中段被忽视 → 避免超长模板，提供精简建议

### 2.4 功能发现设计（2×2 矩阵）

|  | **低系统主动** | **高系统主动** |
|--|--------------|--------------|
| **低用户意图** | 社区/口碑自然传播 | 内容感知气泡提示（慎重，避免打扰） |
| **高用户意图** | 命令触发/搜索，降低摩擦 | 加速用户已在进行的操作 |

**四类发现模式**：
- **输入触发**：`/` 斜杠命令菜单、右键手势、空状态激活（利用"白纸恐惧"时机）
- **上下文感知**：识别内容类型自动弹出建议、识别重复行为提议自动化
- **环境/主动**：异常检测（会议冲突提醒）、预测性浮现（自动翻译外语页面）
- **渐进式**：功能晋级（先建议后进菜单）、情境教程（上传图片时才出现背景移除提示）

## 三、计算阶段设计

### 3.1 等待状态是设计面，不是税负

| 场景类型 | 等待敏感度 | 推荐方案 |
|---------|-----------|---------|
| 实时控制 | 极高（毫秒级） | 流式输出，即时反馈 |
| 事务性操作（支付等） | 高 | 明确进度条，保留状态 |
| 创作/生成 | 中 | 流式显示、分段渐进展示 |
| 对话聊天 | 中低 | 打字指示器即可 |
| 自主后台任务 | 低（可离开） | 完成通知 + 任务摘要 |

**视觉减缓手段**：骨架屏（Skeleton Screen）、局部流式输出、进度指示器靠近用户焦点放置。

### 3.2 智能体场景的进度通信四层

```
通知（Notifications）         ← 用户可离开，被动接收：完成/错误/需审批信号
├── Tier 1: 概览（Overview）  ← 任务进行中？哪个阶段？是否被阻塞？
├── Tier 2: 详情（Detail）    ← 已完成步骤 + 当前进行 + 即将执行
└── Tier 3: 完整记录（Full Record） ← 调试/合规审计，按需展开
```

### 3.3 错误设计

**AI 错误的特殊性**：幻觉错误（内容错误但文字流畅）系统无法自知，检测负担转移到用户。

**五条基本原则**：
1. 可理解：白话说明问题，不用技术代码
2. 可恢复：保存上下文，不从零开始
3. 提前预防可预见错误
4. 优雅降级：部分不可用时提供替代路径
5. 清晰状态：用户随时知道系统处于什么状态

**AI 特有实践**：
- **检查点（Checkpoint）**：像 Git commit，实验前固化稳定状态，支持回滚
- **提示历史**：↑ 箭头调出上次输入，低摩擦重试
- 区分"系统错误"和"输出不满意"，提供不同恢复路径

## 四、输出设计

### 4.1 核心认知：输出不是答案

> **"The output is not the answer."** — 输出是需要被解读、验证、操作的设计产物，不是权威结论。

### 4.2 五大输出设计原则

| 原则 | 核心问题 | 关键设计手段 |
|------|---------|------------|
| **清晰 (Clear)** | 用户能即时理解吗？ | 格式模板、层级标题、简短段落、可扫描结构 |
| **可验证 (Verifiable)** | 用户能核查信息吗？ | 内联引用、来源链接；**不用虚假置信百分比** |
| **有根基 (Grounded)** | 用户知道输出的上下文吗？ | 显示所用模型、地区设定、时间范围、检索来源 |
| **可操作 (Actionable)** | 用户知道下一步做什么吗？ | 明确行动按钮、保存/分享/导出路径 |
| **可调整 (Adjustable)** | 用户能迭代精炼吗？ | 内联精炼选项、版本对比、Canvas 画布模式 |

### 4.3 清晰度：结构化输出模板（可直接用于系统提示词）

```
Format all responses:
  Title: [简洁明确]
  Introduction: [1-2句说明目的]
  Key Concepts: [bullet list 要点]
  Steps: [有序编号步骤]
  Notes/Tips: [可选补充]
  Sources: [带链接的来源]

Rules:
  H2 = 主要章节, H3 = 子章节
  每段最多 2-3 句
  Bold 关键动作/术语
  信息不可用时 → 输出"暂无数据"（不猜测）
  优先可扫描性，而非完整性
```

**注意**：过度模板化会提高幻觉率，让内容显机械。给模型保留一定弹性空间。

### 4.4 可验证性：拒绝假置信度

- **不要**：显示"93% 置信度"——这是下一个 token 的统计概率，不是事实准确率
- **要**：提供可点击的来源链接（Perplexity 角标引用风格）
- **原则**：输出含客观信息且用户会据此行动 → **必须设计人工二次核查层（HITL）**

### 4.5 有根基（Grounding）

Grounding 信号清单（让用户知道输出的来源和上下文）：
- 使用的模型版本
- 所负责的 Agent 或工具名称
- **地理/地区设定**（法规类内容极其重要——加州法与内华达法完全不同）
- 任务模式（创作 vs 分析 vs 草稿 vs 精修）
- 哪条历史对话影响了此回答

推荐模式：`ⓘ 为什么给出这个建议` 信息气泡（点击展开，不打断主流程）。

### 4.6 可操作性与 Canvas 模式

- 提供下游动作按钮（保存/预约/购买/导出/发送）
- 区分"草稿"与"发送"，不可逆操作必须有确认步骤
- **Canvas/画布模式**：可编辑、可分支、版本可回溯的长期工作空间，是输出从"机器生成"转向"人机共创"的关键界面形态

### 4.7 AI 内容披露

EU AI Act（2024）、加州 SB 1001 均要求标注 AI 生成内容。最佳实践：在 AI 生成内容顶部添加清晰标注，视为建立信任的手段，而非警告。

## 五、智能体（Agent）界面设计

### 5.1 五种核心 Agentic 模式

| 模式 | 功能 | 主要界面挑战 |
|------|------|------------|
| **Reflection（反思）** | 呈现前自我评估纠错（generate-reflect-iterate 循环） | 延迟需解释；"已检查"信号建立信任 |
| **Tool Use（工具调用）** | 调用外部 API/数据库/代码执行 | 权限请求设计、访问透明、错误归因 |
| **Planning（规划）** | 将复杂目标分解为有序步骤 | 展示计划 vs 认知负荷的平衡 |
| **Multiagent（多智能体）** | 专业子 Agent 协同，协调者统筹 | 状态看板、冲突可见、汇报归因 |
| **ReAct（推理-行动循环）** | 边推理边行动，根据观察迭代 | 展示迭代次数/思路；不确定时主动升级给人类 |

这些模式**可以组合**：一个系统可同时使用 Planning + Reflection + Tool Use。

### 5.2 三条 Agentic 设计核心原则

#### 原则一：揭示计划（Reveal the Plan）

执行开始之前，先展示 Agent 对任务的解读与执行计划。

```
示范流程（Replit 模式）：
用户提交请求 → Agent 生成计划摘要
→ "我计划：1. 创建3个文件... 2. 实现登录功能... 3. 测试..."
→ 用户确认/修改 → 开始执行
```

- 计划是"合同"：用户批准即约定了范围，事后可对照复盘
- 计划变更时主动通知并说明原因
- 低风险/简单任务可静默规划；复杂/不可逆任务必须展示

#### 原则二：优先呈现最重要的信息（Prioritize What Matters Most）

按任务阶段动态决定优先展示什么：

| 阶段 | 用户最关心的问题 |
|------|----------------|
| 任务刚开始 | Agent 理解我的意图了吗？ |
| 执行中 | 进展正常吗？需要我介入吗？ |
| 任务完成 | 结果是什么？下一步做什么？ |

多智能体场景示例：`"研究 Agent 找到12份资料，分析 Agent 处理中，可视化 Agent 等待数据"` —— 一句话压缩全局状态。

#### 原则三：共享控制（Design for Shared Control）

- **明确边界**：哪些操作 Agent 可自主决定，哪些需用户批准（可配置）
- **随时可暂停**：暂停即刻生效，展示已完成/进行中/待执行，支持从检查点恢复
- **不可逆操作必须确认**（发邮件、修改数据库、删除文件）
- **回滚（Rollback）**：像 Git 版本历史，用户随时可退回任意检查点
- **决策点设计**：将 Agent 暂停转化为自然协作请求（"研究完成，是否继续生成报告？"）

### 5.3 检查点（Checkpoint）设计组成

| 组成元素 | 设计要点 |
|---------|---------|
| **中间输出** | 标注为"初步结果"；用户编辑 = 新输入信号，而非覆盖原有工作 |
| **权限请求** | 说明"访问什么/为什么/如何撤销"；样式统一；支持持久化权限选项 |
| **编辑与错误** | 暂停-调整-继续流程；错误显示在检查点内，不用中断式弹窗 |
| **来源展示** | 列出引用的数据来源，可点击核查，位置固定易找 |

### 5.4 权限分级参考（Claude Code 模式）

| 工具类型 | 示例 | 是否需批准 | "不再询问"行为 |
|---------|------|-----------|--------------|
| 只读操作 | 读文件/搜索 | 否 | N/A |
| Bash 命令/脚本 | 运行脚本 | 是 | 按项目目录+命令永久记忆 |
| 文件写入/修改 | 编辑文件 | 是 | 至 Session 结束 |

## 六、编排与配置（Orchestration）

配置决策应遵循三个元原则：

| 原则 | 含义 |
|------|------|
| **有意图 (Intentional)** | 在有意义的时机呈现选项，让用户理解选择影响 |
| **透明 (Transparent)** | 模型/工具切换时给出可见信号（badge 显示当前模型） |
| **可识别 (Recognizable)** | 用户随时能看出当前状态，清楚如何修改 |

### 核心配置维度

| 维度 | 设计要点 |
|------|---------|
| **模型选择** | 用结果描述代替技术名称（"快速回答" vs "擅长复杂推理"）；智能自动路由 + 显示当前模型 badge |
| **工具管理** | 侧边栏列出 / 上下文自动激活 / 混合；`@web` 语法给高级用户精确控制 |
| **用量计量** | 前置披露配额；用虚拟单位软化金额感；用尽时保留进度支持续接 |
| **权限系统** | 细粒度授权（"仅此对话"/"整个项目"）；撤回入口与授权入口同样易找 |
| **资产库** | 文件生命周期（版本/归档）、跨会话持久化、按内容搜索（不仅按文件名）|

## 七、用户认知偏差与设计应对

| 偏差 | 用户表现 | 设计应对 |
|------|---------|---------|
| **ELIZA 效应** | 将语言流畅误读为真实理解力 | 清晰标注 AI 生成；拟人界面加强披露 |
| **事实性假设** | 把 AI 输出当权威真实信息 | 来源链接 + 核查路径；高风险场景设 HITL |
| **一致性假设** | 以为同样提示总得相同结果 | 解释概率性本质；提供多版本比较 |
| **过度依赖（Automation Bias）** | 无批判接受 AI 建议 | 高风险操作插入认知强制函数 |
| **讨好偏差回路** | AI 放大用户假设，形成信息茧房 | 提供"异见/换角度"入口 |
| **动量行为** | 用户固守路径，忽视更好功能 | 发现引导在正确时机介入，不在专注时打扰 |

## 八、AI 产品组织成熟度模型

| 级别 | 特征 | 典型症状 |
|------|------|---------|
| **Level 1：个人探索** | 各自摸索，功能由高层拍板 | AI 特性"贴上去"，用户困惑；无人负责模型能力传递 |
| **Level 2：共享理解** | 有共同语言和文档，靠手动维护 | 信息流通不稳定；Eval 结果流不到设计侧 |
| **Level 3：集成型 AI 组织** | 信息自动流通；功能源于用户研究 | 清晰知识所有权 + 定期同步机制 |

**自评问题**：谁负责把模型能力变化翻译给设计团队？Eval 结果如何进入设计决策？Responsible AI 指南是否已纳入发布流程？

## 九、快速决策卡片

| 症状 | 检查项 |
|------|--------|
| 用户不知如何使用 AI 功能 | 发现模式是否匹配用户意图强度？空状态是否有引导？Starter Prompt 是否示范场景？ |
| 用户不信任 AI 输出 | 是否附有来源？Grounding 信号是否清晰？高风险操作是否设计了 HITL？ |
| Agent 执行中用户迷失 | 是否在执行前展示了计划？进度通信是否达到用户需要的层级？检查点节奏是否合适？ |
| 用户担心 Agent 不可控 | 是否有随时暂停机制？不可逆操作是否有确认步骤？是否支持回滚？ |
| AI 输出质量不稳定 | 系统提示词是否定义了格式模板？直接操控是否分担了提示词负担？CARE 四要素是否通过 UI 引导？ |
| 模型更新导致用户困惑 | 是否建立了模型能力变更的传递机制？产品内是否有变更提示？ |

## 十、关键概念词汇表

| 术语 | 含义 |
|------|------|
| ELIZA 效应 | 将语言流畅性误读为真实理解力的用户倾向 |
| Sycophancy | 模型倾向顺从用户假设而非纠正的偏差 |
| Grounding | 输出声明其上下文（来源/模型/地区/时间）的设计实践 |
| HITL | Human-in-the-Loop，高风险输出必须经人工核查的机制 |
| Checkpoint | Agent 长任务中主动暂停展示进度并请求确认的时刻 |
| Rollback | 回滚至上一检查点的能力，类比 Git 版本历史 |
| Canvas | 可编辑/可分支/可回溯的 AI 生成内容工作空间 |
| ReAct | Reason and Act，推理与行动交替迭代的 Agent 模式 |
| Reflection | Agent 自我评估纠错后再呈现结果的机制 |
| ICO 框架 | Input-Computation-Output，本书核心分析框架 |
| CARE 框架 | Context-Action-Results-Examples，结构化提示方法 |
| Momentum Behavior | 用户固守当前路径、忽视更好功能的心理现象 |
